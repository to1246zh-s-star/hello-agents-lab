## 1. 核心概念:什么是上下文工程

**一句话区分**:Prompt Engineering关心"这句话怎么措辞",Context Engineering关心"这一刻上下文窗口里该摆哪些信息、按什么顺序摆、摆多少"。

**核心比喻**:上下文窗口是一张有限大小的办公桌,不是仓库。桌上堆得越乱,越容易"分心"(Distraction)、放错文件会"投毒"后续推理(Poisoning)、放了不该放的说明书会"混淆"(Confusion)、放了矛盾的文件会"冲突"(Clash)。

**书本(9.1-9.2节)给出的应对策略**,对应到2026年业界通用的`write/select/compress/isolate`四件套:

| 书本术语 | 业界通用术语 | 落地组件 |
|---|---|---|
| 结构化笔记 | write | NoteTool |
| (GSSC的Select阶段) | select | ContextBuilder._select() |
| 压缩整合 | compress | ContextBuilder._compress() |
| 子代理架构 | isolate | 书本只有理论,本手册补上代码 |

### 3.2 ⚠️ 实际踩坑记录:你刚遇到的问题

你执行了:
```bash
root@autodl-container-xxx:~/mini_context# python3 context_builder.py
root@autodl-container-xxx:~/mini_context# python3 context_builder.py
```
两次都没有任何输出。**这不是报错**——`context_builder.py`这个文件本身只定义了`ContextPacket`、`ContextConfig`、`ContextBuilder`三个类,没有任何会打印内容的顶层代码,所以`python3 context_builder.py`执行完就静默退出,返回码0,这是正常行为。

更早之前你还遇到过把Python代码整段粘贴进bash终端执行的情况(报了一堆`command not found`),根因是**把.py文件内容当shell命令执行了**,而不是用`python3 文件名.py`来跑。

**结论**:定义类/函数的文件本身不会有输出,必须跑对应的demo/测试脚本才能看到效果。记住这个原则,后面每一步我都会明确告诉你该跑哪个文件。

### 3.4 书本实现里发现的两个真实问题(已验证,不是猜测)

1. **`_calculate_relevance()`用Jaccard(按空格分词)算相关性,对中文基本失效**——中文句子没有空格,`content.lower().split()`切不出有意义的词,和query几乎不产生交集。实测中两条明显相关的中文记忆被`min_relevance`阈值误杀,根本没进最终上下文。
2. **`[Context]`区的顺序会被打乱**——因为Select阶段按`combined_score`降序贪心选择,不是按对话时间顺序,实测输出里`assistant`的回复排在了更早的`user`发言前面。
3. （额外发现,书本代码本身的含糊点）**`ContextConfig.reserve_ratio`字段在文档里说要"为系统指令预留空间",但翻遍`_select()`实现根本没用上**——系统指令按实际消耗直接扣除,不是按比例预留池子。这是书本设计文档和代码实现对不上的地方,已如实记录,不擅自"修正"。

## 6. 五项现代升级

### 6.1 升级1:relevance从Jaccard换成TF-IDF

**动机**:第3.4节发现的中文相关性失效问题。**做法**:继承`ContextBuilder`,只覆写`_calculate_relevance`,书本原始类完整保留可对照。

```bash
python3 -c "
from relevance_scorer import TfidfContextBuilder
from context_builder import ContextConfig
from mock_tools import MockMemoryTool, MockRAGTool

memory_tool = MockMemoryTool()
rag_tool = MockRAGTool()
memory_tool.run({'action':'add','content':'用户正在开发数据分析工具,使用Python和Pandas','importance':0.8})
memory_tool.run({'action':'add','content':'已完成CSV读取模块的开发','importance':0.7})

config = ContextConfig(max_tokens=3000, min_relevance=0.05, enable_compression=True)
builder = TfidfContextBuilder(memory_tool=memory_tool, rag_tool=rag_tool, config=config)
print(builder.build(user_query='如何优化Pandas的内存占用?', system_instructions='你是数据工程顾问'))
"
```

**实测验证**:换成TF-IDF后,`"用户正在开发数据分析工具,使用Python和Pandas"`(因为和query共享"Pandas"字符片段)成功被选入`[Context]`区;`"已完成CSV读取模块的开发"`(无关)正确被过滤。**从"完全失效"到"基本可用"的真实提升**。

**继续升级的方向**(需要联网,建议在AutoDL上做,本地沙箱环境无法验证):换成`BAAI/bge-small-zh-v1.5`这类中文embedding模型,解决"用词不同但语义相同"这类TF-IDF依然搞不定的情况(比如"省内存" vs "降低内存占用")。

### 6.2 升级2:上下文质量评估(书本习题2第二问)

```bash
python3 -c "
from context_quality import ContextQualityEvaluator
from relevance_scorer import TfidfContextBuilder
from context_builder import ContextConfig
from mock_tools import MockMemoryTool, MockRAGTool
import json

memory_tool = MockMemoryTool()
rag_tool = MockRAGTool()
memory_tool.run({'action':'add','content':'用户正在开发数据分析工具,使用Python和Pandas','importance':0.8})

config = ContextConfig(max_tokens=3000, min_relevance=0.05)
builder = TfidfContextBuilder(memory_tool=memory_tool, rag_tool=rag_tool, config=config)
query = '如何优化Pandas的内存占用?'
ctx = builder.build(user_query=query, system_instructions='你是数据工程顾问')

evaluator = ContextQualityEvaluator()
report = evaluator.evaluate(ctx, query, config)
print(json.dumps(report, ensure_ascii=False, indent=2))
"
```

**三个评估维度**:信息密度(`[Context]`区是否有重复行)、相关性(`[Evidence]+[Context]`和query的字面重合度)、完整性(必要分区`[Task]/[Output]`是否缺失,建议分区`[Evidence]/[Context]`是否缺失)。

**实测中一个真实案例**:后面第6.6节完整集成测试时,评估器准确捕捉到了"笔记应该被召回但没有被召回"的情况,`completeness`维度报告缺少`[Context]`分区并给出"可能是检索没召回任何内容"的建议——证明这个评估器不是摆设,真能揪出GSSC流水线的失灵。

### 6.3 升级3:子代理隔离(补上书本9.2.3节缺失的代码)

**动机**:书本9.2.3节明确描述了子代理架构("主代理负责规划,子代理在干净上下文里深挖,只回传1000-2000 tokens摘要"),但9.3-9.5节的三个组件里没有一个是子代理实现。

**关键设计**:子代理摘要写入NoteTool持久化,而不是只留在Python变量里——这样主代理即使发生一次上下文重置(Compaction),依然能从笔记里找回结论。

这一步**需要真实LLM调用**,本地沙箱没有API Key无法验证,必须在AutoDL上跑:

```bash
python3 -c "
from sub_agent import MainAgentOrchestrator
from note_tool import NoteTool

notes = NoteTool(workspace='./sub_agent_notes')
orchestrator = MainAgentOrchestrator(note_tool=notes)

subtasks = [
    {'description': '分析src/utils.py的代码复杂度', 'evidence': '这里粘贴TerminalTool读到的文件内容'},
    {'description': '检查是否存在未处理的异常', 'evidence': '这里粘贴grep结果'},
]
results = orchestrator.dispatch(subtasks)
for r in results:
    print(r['subtask'], '->', r['summary'][:100])
"
```

**建议动手做的对比实验**(对应本手册习题2):统计"隔离模式"(主线程只留摘要) vs "广播模式"(主线程留全部执行细节)两种方式下,主线程最终token占用的差异,给出具体节省比例。

### 6.4 升级4:多智能体上下文路由(应对Broadcaster反模式)

**动机**:2026年多篇上下文工程综述提到的常见错误——把同一份完整上下文原样广播给流水线里每个子Agent。

```bash
python3 -c "
from context_router import RoleBasedContextRouter
from relevance_scorer import TfidfContextBuilder
from mock_tools import MockMemoryTool, MockRAGTool

router = RoleBasedContextRouter(TfidfContextBuilder, memory_tool=MockMemoryTool(), rag_tool=MockRAGTool())
router.register_role('explorer', max_tokens=1500, use_memory=False, use_rag=True, system_instructions='你负责探索代码库,只关注技术证据')
router.register_role('summarizer', max_tokens=800, use_memory=True, use_rag=False, system_instructions='你负责总结进展,只关注历史记忆')

for role in ['explorer', 'summarizer']:
    r = router.route_report(role, '如何优化Pandas内存占用?')
    print(role, '-> tokens:', r['tokens'])
"
```

**实测验证**:`explorer`角色(只开RAG)拿到121 tokens、包含`[Evidence]`区;`summarizer`角色(只开memory)拿到48 tokens、完全没有`[Evidence]`区。两个角色确实拿到了不同的定制化上下文,而不是同一份广播内容。

### 6.5 升级5:Prompt Caching感知的上下文排布

**动机**:书本GSSC流水线固定输出`[Role&Policies]→[Task]→[Evidence]→[Context]→[Output]`,但`[Task]`(每次query都变)排在第二位,导致即使`[Evidence]`和`[Context]`没变,也无法复用缓存前缀。

```bash
python3 -c "
from relevance_scorer import TfidfContextBuilder
from context_builder import ContextConfig
from mock_tools import MockMemoryTool, MockRAGTool
from prompt_cache_layout import reorder_for_prompt_caching

builder = TfidfContextBuilder(memory_tool=MockMemoryTool(), rag_tool=MockRAGTool(), config=ContextConfig(max_tokens=3000))
ctx = builder.build(user_query='如何优化Pandas的内存占用?', system_instructions='你是数据工程顾问')
result = reorder_for_prompt_caching(ctx)
print('--- cacheable_prefix(稳定,可复用) ---')
print(result['cacheable_prefix'])
print('--- dynamic_suffix(每次必变) ---')
print(result['dynamic_suffix'])
"
```

## 7. AutoDL完整踩坑记录(累积自本次对话全程)

| 坑 | 原因 | 解决方案 |
|---|---|---|
| **`python3 context_builder.py`没有任何输出** | 该文件只定义类,没有顶层可执行代码,这是正常行为 | 要跑`demo_context_builder.py`或写测试代码调用类,不能指望"定义类的文件"本身有输出 |
| **把.py文件内容粘贴进bash终端执行,报一堆`command not found`** | 把Python代码当shell命令解析了 | 用`cat > file.py << 'EOF'`创建文件,或用编辑器保存,然后固定用`python3 file.py`执行,不要`sh file.py`或直接粘贴到终端 |
| TF-IDF在极短文本上报`ValueError: empty vocabulary` | char ngram在文本长度小于ngram下限时词表为空 | `_calculate_relevance`里用try-except兜底返回0.0 |
| `min_relevance`阈值设太高,笔记/记忆大量被过滤 | 书本默认`min_relevance=0.1`是给Jaccard这种粗糙打分方式设的,换TF-IDF后分数分布区间不同 | 换打分方式后要重新校准阈值,先跑`context_quality.py`观察实际分数分布再定阈值 |
| 多智能体路由测试时每个角色都new了一个工具实例 | 忘记在`RoleBasedContextRouter`外层复用同一个工具实例 | 工具实例在Router外层初始化一次,传引用进去 |
| BGE中文embedding模型下载超时 | AutoDL访问ModelScope/HuggingFace有时不稳定 | 优先用ModelScope镜像下载,或先用TF-IDF版本跑通全流程,embedding作为后续优化不阻塞主线 |

# 第九章:上下文工程 —— 对齐《HelloAgents》书本的重写版

> 上一版`mini_context/`是我自己按通用上下文工程理论设计的(budget_manager/compactor/scratchpad/tool_retriever/sub_agent_isolation)。
> 现在拿到了书里9.3-9.5节的真实代码,发现两者**不是一回事**,这一版彻底推翻重来,逐行对照书本实现,不做任何"自己觉得更好"的擅自修改。

---

## 0. 先说清楚:新旧版本的映射关系

| 旧版模块(我自己发挥的) | 新版对应 | 结论 |
|---|---|---|
| `budget_manager.py`(配额制:history和context_chunks各占一个独立预算池) | `ContextBuilder._select()`(统一排序制:所有非系统指令packet混在一起按分数排序) | **设计哲学完全不同,不是简化关系,是替换关系** |
| `compactor.py`(预防性,历史超过阈值主动调LLM摘要) | `ContextBuilder._compress()`(兜底性,只有最终文本超预算才做,而且是简单截断不调LLM) | 书里的压缩发生的时机、触发条件、实现方式都不同,直接废弃旧版 |
| `scratchpad.py`(内存中的字典,精确key-value) | `NoteTool`(Markdown文件+YAML元数据,持久化到磁盘) | 书里的笔记本是真正的文件系统持久化,不是进程内字典,废弃旧版 |
| `tool_retriever.py`(TF-IDF检索工具说明书) | **书里没有对应组件** | 书里9.5节的`TerminalTool`解决的是完全不同的问题(安全的文件系统访问),不是"从工具池里挑工具"。这个模块是我基于通用理论加的,和教材9.3-9.5节没有关系,予以移除 |
| `sub_agent_isolation.py`(独立LLM调用子任务) | **书里没有对应组件** | 书里9.2.3节只在理论层面讨论了子代理架构,没有给出配套代码。这个模块也移除 |

**结论**:这次重写只保留三个文件,一一对应书里9.3、9.4、9.5节的三个真实组件:

```
mini_context/
├── context_builder.py   # 对应9.3节 ContextBuilder(GSSC流水线)
├── note_tool.py         # 对应9.4节 NoteTool(结构化笔记)
├── terminal_tool.py     # 对应9.5节 TerminalTool(沙箱终端)
├── mock_tools.py         # 仅用于本地验证,接口对齐第八章MemoryTool/RAGTool
└── demo_context_builder.py
```

## 1. ContextBuilder:逐行对照书里9.3节

### 1.1 两个数据结构,严格照抄

`ContextPacket`(候选信息包)和`ContextConfig`(配置)两个`dataclass`,字段、默认值、`__post_init__`里的校验逻辑,全部和书里一致,一个字段都没改。唯一要理解的细节:

- `ContextPacket.relevance_score`默认是`0.5`——这个默认值在`_select()`里有特殊含义:**等于0.5就意味着"还没算过相关性,需要现算"**。这是一个隐式约定,不是文档里明说的,读代码才能发现。

### 1.2 GSSC四阶段,和我理解的不一样的地方

**Gather阶段**:书里的做法是"自己主动去调`memory_tool.run()`和`rag_tool.run()`",而不是像我旧版那样"假设外部已经把排好序的结果传进来"。这意味着`ContextBuilder`是**主动的检索发起者**,不是被动的"格式转换器"。这是本质区别——旧版的`TokenBudgetManager`根本不做检索这件事。

**Select阶段**(这是和旧版分歧最大的地方):

```python
combined_score = self.config.relevance_weight * packet.relevance_score + self.config.recency_weight * recency
```

所有非系统指令的packet(不管来自memory、RAG还是历史对话)**混在一起统一按这一个分数排序**,然后贪心装箱直到预算耗尽。这意味着:

- 历史对话和检索结果是"公平竞争"的,没有"历史至少保留几条"这种保底机制
- 如果一条历史消息文字上和当前query毫无重合、时间也比较久,它完全可能被完全挤出上下文——这是我上一版配额制`budget_manager.py`做不到的行为

**relevance计算方式是Jaccard(关键词重叠),不是向量相似度**。我在验证时发现了一个值得记录的真实问题(见第4节)。

**Structure阶段**:固定输出`[Role & Policies] / [Task] / [Evidence] / [Context] / [Output]`五个分区,分类规则很简单粗暴——`metadata.type`是`rag_result`或`knowledge`就进`[Evidence]`,其余全部进`[Context]`。这意味着**记忆(memory)和历史对话(conversation_history)会被分到同一个`[Context]`区,没有区分**。

**Compress阶段**:只有`_structure()`产出的最终文本仍然超过`max_tokens`时才触发,做法是按`\n\n`切分区块,从前往后能装的整段装,装不下的那个区块做字符比例截断,后面的区块直接丢弃。**这是纯截断,不调用LLM做摘要**——和我上一版`compactor.py`用LLM生成摘要是完全不同的两种"压缩"。

### 1.3 一个书本代码里含糊的地方(诚实记录,不擅自"修正")

`ContextConfig.reserve_ratio`(为系统指令预留的比例)这个字段,在9.3.2节的注释里说是"确保系统指令等关键信息始终有足够的空间",但翻遍9.3.3节给出的`_select()`实现,**这个字段实际上完全没被用到**——系统指令占用的token是按它"实际消耗多少"直接从`available_tokens`里扣除的,不是按`reserve_ratio`预留一个固定比例的池子。也就是说如果系统指令很短,`reserve_ratio=0.2`不会额外保留20%的空间给它;如果系统指令很长,超过20%也不会被拦截。

我在`context_builder.py`里保留了这个字段(因为要保持接口和书里一致),但在代码注释里诚实标注了这个"设计文档说一套、实现做一套"的现象,没有擅自加代码去"修正"它——这是这本书9.3节代码本身的一个可以在课堂/面试里讨论的点,不是我们实现错了。

## 2. NoteTool:逐行对照书里9.4节

七个操作(`create/read/update/search/list/summary/delete`)全部照抄,存储格式是`---\nYAML\n---\n\nMarkdown正文`,配合一个`notes_index.json`做索引,一个字段都没改。

验证时打印出的真实文件长这样(完全符合书里9.4.2节展示的格式):

```yaml
---
id: note_20260714_093137_0
title: 重构项目-第一阶段
type: task_state
tags:
- refactoring
- phase1
created_at: '2026-07-14T09:31:37.818570'
updated_at: '2026-07-14T09:31:37.819033'
---

## 完成情况
已完成数据模型层重构,测试覆盖率85%。
```

这个和我上一版`scratchpad.py`的核心区别是:**NoteTool是真正持久化到磁盘的独立文件**,重启进程、换一台机器,只要`workspace`目录还在,笔记就还在;而旧版`scratchpad.py`本质是一个内存字典,虽然我也写了`persist_path`做JSON持久化,但那是我自己加的,书里的设计从一开始就是"一条笔记=一个独立.md文件",不是"整个笔记本=一个JSON blob"。**独立文件的好处是天然支持Git版本控制**——这是书里9.4.1节强调的"版本友好"这个设计目标,我旧版实现完全没考虑到这一点。

四层安全机制照抄:命令白名单(`ALLOWED_COMMANDS`集合)、工作目录沙箱(`Path.relative_to()`检测越权)、超时控制(`subprocess.run(timeout=...)`)、输出大小限制。

## 4. 验证时发现的真实问题(诚实记录,不回避)

### 4.1 Jaccard相关性对中文几乎失效

`_calculate_relevance()`用的是`content.lower().split()`按空格分词再算Jaccard相似度。**中文句子内部没有空格**,`"用户正在开发数据分析工具,使用Python和Pandas".split()`基本上会被切成一整个大"词"(或者极少数几个),和query`"如何优化Pandas的内存占用?"`按空格分词后几乎不会有交集(除了"Pandas"这个英文词偶然重合)。

实测结果验证了这一点:demo跑出来,两条中文记忆(`"用户正在开发数据分析工具..."`和`"已完成CSV读取模块的开发"`)在`_select()`里因为`relevance_score`算出来低于`min_relevance=0.2`阈值,**直接被过滤掉了,根本没进入最终上下文**——尽管这两条记忆在语义上明显与"Pandas内存优化"这个话题相关。这不是bug,是书里"关键词重叠"这个简化实现对中文场景的真实局限,9.3.5节的"最佳实践"里其实也提到了"生产环境应替换为向量相似度计算",算是书本自己也承认的已知局限。

### 4.2 `[Context]`区的顺序会被打乱,不是原始时间顺序

因为`_select()`是按`combined_score`降序贪心选择,**历史对话的先后顺序在这一步被打散了**,最后进`[Context]`区的顺序是"分数从高到低",不是"对话发生的先后顺序"。实测输出里能看到`assistant`的回复排在了更早的`user`发言前面。这在阅读体验上是反直觉的——人类习惯按时间顺序读对话,但这套实现优先保证"分数高的排前面"。这也是一个可以在课堂/面试讨论的设计取舍:GSSC为了"信息密度优先"牺牲了"叙事连贯性"。
